# Redis 자료구조 완전 정복 — Spring Data Redis 실습

> **시리즈**: Redis 학습 정리  
> **환경**: Spring Boot 4.0.5 / Spring Data Redis 4.0.4 / Java 21 / Docker Redis (localhost:6379)

---

## 들어가며

Redis는 단순한 캐시가 아닙니다. 다양한 자료구조를 제공하고, 각 자료구조마다 특화된 커맨드가 있어 상황에 맞게 활용할 수 있습니다.

이 글에서는 Redis가 제공하는 **9가지 자료구조**와 **키 관리 커맨드**를 Spring Data Redis 코드 중심으로 정리합니다. Redis CLI 예제는 동작 원리 이해를 위한 참고로 함께 제공합니다.

Spring Data Redis는 `RedisTemplate`을 통해 자료구조별 Operations 객체를 제공합니다.

```java
redisTemplate.opsForValue()       // String
redisTemplate.opsForList()        // List
redisTemplate.opsForHash()        // Hash
redisTemplate.opsForSet()         // Set
redisTemplate.opsForZSet()        // Sorted Set
redisTemplate.opsForHyperLogLog() // HyperLogLog
redisTemplate.opsForGeo()         // Geospatial
redisTemplate.opsForStream()      // Stream
// Bitmap은 opsForValue() 에서 bit 관련 메서드 사용
```

## 1. String

가장 기본적인 자료구조로, 문자열뿐만 아니라 **숫자 값도 저장하고 원자적으로 조작**할 수 있습니다.

### SET / GET

`opsForValue()`가 반환하는 `ValueOperations`를 통해 사용합니다.

```java
// SET hello world
opsForValue().set("hello", "world");

// GET hello → "world"
String value = opsForValue().get("hello");
```

### NX / XX 옵션

`setIfAbsent`와 `setIfPresent`로 조건부 저장을 처리합니다. 반환값은 실제로 저장이 이루어졌는지 여부(`Boolean`)입니다.

```java
// SET hello newval NX → 키가 없을 때만 저장
// 이미 키가 존재하면 false 반환 (저장 안됨)
Boolean isSet = opsForValue().setIfAbsent("hello", "newval");

// SET hello newval XX → 키가 있을 때만 덮어씀
// 키가 없으면 false 반환 (저장 안됨)
Boolean isUpdated = opsForValue().setIfPresent("hello", "newval");
```

> **Redis CLI 참고**
> ```bash
> > SET hello newval NX  →  (nil)   # 키 있으면 실패
> > SET hello newval XX  →  OK      # 키 있으면 성공
> ```

### 숫자 조작

`increment` / `decrement`는 **원자적(atomic)** 으로 동작하므로 동시성 문제 없이 카운터를 구현할 수 있습니다.

```java
opsForValue().set("counter", "100");

Long v1 = opsForValue().increment("counter");        // INCR   → 101
Long v2 = opsForValue().increment("counter", 50L);   // INCRBY → 151
Long v3 = opsForValue().decrement("counter");        // DECR   → 150
Long v4 = opsForValue().decrement("counter", 50L);   // DECRBY → 100
```

### 다중 키 조작

`multiSet` / `multiGet`은 여러 키를 **단일 네트워크 왕복**으로 처리합니다. 대규모 시스템에서 성능 이점이 큽니다.

```java
// MSET a 10 b 20 c 30
opsForValue().multiSet(Map.of("a", "10", "b", "20", "c", "30"));

// MGET a b c → ["10", "20", "30"]
// 키 순서대로 반환, 없는 키는 null
List<String> values = opsForValue().multiGet(List.of("a", "b", "c"));
```

## 2. List

**순서가 있는 문자열 목록**으로 스택(Stack)과 큐(Queue) 모두 구현 가능합니다. `opsForList()`가 반환하는 `ListOperations`를 사용합니다.

### 데이터 추가

```java
// LPUSH - 왼쪽(head)에 추가, 반환값은 추가 후 전체 리스트 길이
Long size = opsForList().leftPushAll("mylist", "D", "A", "C", "B", "A");

// RPUSH - 오른쪽(tail)에 추가
opsForList().rightPushAll("mylist", "B");
```

> **Redis CLI 참고** — 위 코드와 동일한 결과
> ```bash
> > LPUSH mylist E → [E]
> > RPUSH mylist B → [E, B]
> > LPUSH mylist D A C B A → [A, B, C, A, D, E, B]
> ```

### 조회

```java
// LRANGE - 인덱스 범위 조회, -1은 마지막 인덱스
List<String> all  = opsForList().range("mylist", 0, -1);  // 전체
List<String> part = opsForList().range("mylist", 0, 3);   // 앞 4개

// LINDEX - 특정 인덱스 조회
String item = opsForList().index("mylist", 3);
```

### 꺼내기 (LPOP)

`leftPop`은 리스트에서 꺼내는 동시에 **삭제**합니다.

```java
String one = opsForList().leftPop("mylist");           // 1개 꺼내기
List<String> many = opsForList().leftPop("mylist", 2); // 2개 꺼내기
```

### LTRIM — 고정 길이 큐

지정한 범위 **밖의 아이템을 모두 삭제**합니다. 반환값은 없습니다.

`LPUSH` + `LTRIM` 조합으로 최신 N개만 유지하는 고정 길이 큐를 구현할 수 있습니다.

```java
// 최근 1000개만 유지하는 로그 큐
opsForList().leftPush("logdata", data);
opsForList().trim("logdata", 0, 999);
```

### 삽입 / 수정

```java
// LINSERT BEFORE - pivot 앞에 삽입
opsForList().leftPush("mylist", "B", "E");   // B 앞에 E 삽입

// LINSERT AFTER - pivot 뒤에 삽입
opsForList().rightPush("mylist", "B", "E");  // B 뒤에 E 삽입

// LSET - 인덱스 위치 덮어씀 (범위 초과 시 예외 발생)
opsForList().set("mylist", 2, "F");
```

## 3. Hash

**키 → (필드 - 값)** 구조로 객체를 저장하기에 적합합니다. 필드와 값 모두 String입니다. `opsForHash()`가 반환하는 `HashOperations`를 사용합니다.

> **키 네이밍 관례**  
> `Product:123`, `user:100:cart`처럼 `:` 를 구분자로 사용해 네임스페이스를 표현하는 것이 Redis 관례입니다.  
> `:` 자체는 특별한 기능이 없는 일반 문자이지만, `SCAN` 시 `Product:*` 패턴 조회를 편리하게 해줍니다.

### 저장

```java
// HSET - 단일 필드 저장
opsForHash().put("Product:123", "Name", "Happy Hacking");
opsForHash().put("Product:123", "TypeID", "35");

// HSET - 여러 필드 한 번에 저장 (putAll → HSET 다중)
opsForHash().putAll("Product:234", Map.of(
    "Name", "Track Ball",
    "TypeID", "32"
));
```

### 조회

```java
// HGET - 단일 필드 조회
Object typeId = opsForHash().get("Product:123", "TypeID"); // "35"

// HMGET - 여러 필드 한 번에 조회, 없는 필드는 null 반환
List<Object> fields = opsForHash().multiGet("Product:234",
    List.of("Name", "TypeID"));  // ["Track Ball", "32"]

// HGETALL - 모든 필드-값 쌍 반환
Map<Object, Object> all = opsForHash().entries("Product:234");
```

### HSCAN

대량의 필드를 가진 Hash를 안전하게 순회할 때 사용합니다. `KEYS`와 같은 이유로 `HGETALL` 대신 권장됩니다.

```java
ScanOptions options = ScanOptions.scanOptions().match("*ID*").build();
try (Cursor<Map.Entry<Object, Object>> cursor = opsForHash().scan("Product:123", options)) {
    cursor.forEachRemaining(entry ->
        System.out.println(entry.getKey() + " = " + entry.getValue())
    );
}
```

> **Redis CLI 참고**
> ```bash
> > HGET Product:123 TypeID     → "35"
> > HMGET Product:234 Name TypeID → ["Track Ball", "32"]
> > HGETALL Product:234         → {Name: Track Ball, TypeID: 32}
> ```

## 4. Set

**정렬되지 않은 문자열 모음**으로 중복을 자동으로 제거합니다. `opsForSet()`이 반환하는 `SetOperations`를 사용합니다.

### 추가 / 조회 / 삭제

```java
// SADD - 중복은 무시, 반환값은 실제로 추가된 원소 수
Long added = opsForSet().add("myset", "A", "A", "B", "C"); // → 3 (A 중복 무시)

// SMEMBERS - 전체 조회 (순서 보장 안됨)
Set<String> members = opsForSet().members("myset");

// SREM - 특정 원소 삭제
opsForSet().remove("myset", "B");

// SPOP - 랜덤 원소 반환 + 삭제
String popped = opsForSet().pop("myset");
```

### 집합 연산

```java
// SINTER - 교집합
Set<String> inter = opsForSet().intersect("set:111", "set:222");

// SUNION - 합집합
Set<String> union = opsForSet().union("set:111", "set:222");

// SDIFF - 차집합 (첫 번째 키 기준)
Set<String> diff = opsForSet().difference("set:111", "set:222");
```

> **Redis CLI 참고**
> ```bash
> > SADD set:111 A B C D E
> > SADD set:222 D E F G H
> > SINTER set:111 set:222  → {D, E}
> > SUNION set:111 set:222  → {A,B,C,D,E,F,G,H}
> > SDIFF  set:111 set:222  → {A, B, C}
> ```

### SSCAN

대량의 원소를 가진 Set을 안전하게 순회할 때 사용합니다.

```java
ScanOptions options = ScanOptions.scanOptions().match("user:*").build();
try (Cursor<String> cursor = opsForSet().scan("myset", options)) {
    cursor.forEachRemaining(member -> System.out.println(member));
}
```

## 5. Sorted Set

**스코어(score)를 기준으로 정렬된 집합**입니다. `opsForZSet()`이 반환하는 `ZSetOperations`를 사용합니다.

- 중복 없음 → Set과 유사
- 각 아이템에 score 연결 → Hash와 유사
- score 기준 정렬, 인덱스 접근 가능 → List와 유사
- **score가 같으면 사전 순으로 정렬**

### 저장 (ZADD)

```java
// 기본 저장 - 반환값은 새로 추가된 원소면 true, 기존 원소 업데이트면 false
Boolean isNew = opsForZSet().add("score:220817", "user:B", 100);

// NX - 존재하지 않을 때만 삽입 (기존 score 업데이트 X)
opsForZSet().addIfAbsent("score:220817", "user:B", 999);

// XX / LT / GT 는 Spring Data Redis에서 직접 지원하지 않으므로
// score 조회 후 조건 처리
Double current = opsForZSet().score("score:220817", "user:B");
if (current == null || newScore < current) {           // LT
    opsForZSet().add("score:220817", "user:B", newScore);
}
```

### 인덱스 기반 조회

`rangeWithScores`는 `TypedTuple` 컬렉션을 반환하며, `.getValue()`로 멤버, `.getScore()`로 score를 가져올 수 있습니다.

```java
// ZRANGE 1 3 WITHSCORES - 오름차순
Set<ZSetOperations.TypedTuple<String>> asc =
    opsForZSet().rangeWithScores("score:220817", 1, 3);

// ZRANGE 1 3 WITHSCORES REV - 내림차순
Set<ZSetOperations.TypedTuple<String>> desc =
    opsForZSet().reverseRangeWithScores("score:220817", 1, 3);

asc.forEach(tuple ->
    System.out.println(tuple.getValue() + ": " + tuple.getScore())
);
```

### 스코어 기반 조회 (BYSCORE)

`Double.POSITIVE_INFINITY` / `Double.NEGATIVE_INFINITY`로 Redis의 `+inf` / `-inf`를 표현합니다.

```java
// 100 이상 150 이하
Set<ZSetOperations.TypedTuple<String>> result =
    opsForZSet().rangeByScoreWithScores("score:220817", 100, 150);

// 200 이상 +inf (상위 score 전체)
Set<ZSetOperations.TypedTuple<String>> top =
    opsForZSet().rangeByScoreWithScores("score:220817", 200, Double.POSITIVE_INFINITY);

// 역순 (높은 score 순)
Set<ZSetOperations.TypedTuple<String>> rev =
    opsForZSet().reverseRangeByScoreWithScores("score:220817", 200, Double.POSITIVE_INFINITY);
```

> **Redis CLI 참고** — `(` 접두사는 해당 값을 포함하지 않음(exclusive)
> ```bash
> > ZRANGE score:220817 100 150 BYSCORE WITHSCORES
> → user:B(100), user:A(150), user:C(150)
>
> > ZRANGE score:220817 (100 150 BYSCORE WITHSCORES
> → user:A(150), user:C(150)   ← 100 제외
> ```

### 사전 순 조회 (BYLEX)

score가 모두 동일할 때 유효합니다. `Range.Bound`로 경계 포함 여부를 지정합니다.

```java
// Redis: ZRANGE mySortedSet (b (f BYLEX  → b 초과 f 미만
Set<String> result = opsForZSet().rangeByLex("mySortedSet",
    Range.of(
        Range.Bound.exclusive("b"),   // ( → exclusive (초과)
        Range.Bound.exclusive("f")    // ( → exclusive (미만)
    )
);
// 결과: [banana, candy, dream, egg]
```

Redis 표기와 Spring 표기 대응은 다음과 같습니다.

| Redis | Spring |
|-------|--------|
| `(value` (exclusive, 미포함) | `Range.Bound.exclusive("value")` |
| `[value` (inclusive, 포함) | `Range.Bound.inclusive("value")` |

### ZSCAN

```java
ScanOptions options = ScanOptions.scanOptions().match("user:*").build();
try (Cursor<ZSetOperations.TypedTuple<String>> cursor = opsForZSet().scan("score:220817", options)) {
    cursor.forEachRemaining(tuple ->
        System.out.println(tuple.getValue() + ": " + tuple.getScore())
    );
}
```

## 6. Bitmap

**String + 비트 연산**의 조합입니다. `opsForValue()`의 bit 관련 메서드를 사용합니다.

사용자 수천만 명의 일별 출석 체크처럼 **대량의 boolean 플래그**를 관리할 때 Set보다 수십 배 적은 메모리를 사용합니다.

### SETBIT / GETBIT

```java
// SETBIT - 특정 오프셋에 비트 설정, 반환값은 이전 비트값
Boolean previous = opsForValue().setBit("mybitmap", 2, true);  // → false (이전 값)

// GETBIT - 특정 오프셋 비트 조회
Boolean bit = opsForValue().getBit("mybitmap", 2);  // → true
```

### BITFIELD

여러 오프셋을 **한 번의 커맨드**로 설정할 때 사용합니다. `BitFieldSubCommands`를 조합해 명령을 구성합니다.

```java
// BITFIELD mybitmap SET u1 6 1 SET u1 10 1 SET u1 14 1
// u1 = unsigned 1bit 타입
BitFieldSubCommands cmd = BitFieldSubCommands.create()
    .set(BitFieldSubCommands.BitFieldType.unsigned(1)).valueAt(6L).to(1L)
    .set(BitFieldSubCommands.BitFieldType.unsigned(1)).valueAt(10L).to(1L)
    .set(BitFieldSubCommands.BitFieldType.unsigned(1)).valueAt(14L).to(1L);

List<Long> results = redisTemplate.execute(
    (RedisCallback<List<Long>>) conn ->
        conn.stringCommands().bitField("mybitmap".getBytes(), cmd)
);
```

### BITCOUNT

```java
// BITCOUNT - 1로 설정된 비트 수 반환
Long count = redisTemplate.execute(
    (RedisCallback<Long>) conn ->
        conn.stringCommands().bitCount("mybitmap".getBytes())
);
// offset 2, 6, 10, 14 → 4
```

> `BITFIELD`와 `BITCOUNT`는 `ValueOperations`에서 직접 지원하지 않으므로 `RedisCallback`을 통해 커넥션 레벨에서 호출합니다.

## 7. HyperLogLog

**집합의 카디널리티(고유 원소 수)를 추정**하는 자료구조입니다. `opsForHyperLogLog()`를 사용합니다.

- Set과 달리 원소 자체를 저장하지 않습니다.
- 메모리 최대 **12KB**만 사용합니다.
- 약 **0.81% 오차율**을 허용합니다.

"일별 순 방문자 수(UV)"처럼 **정확한 수보다 추정치로 충분한 경우**에 메모리를 크게 절약할 수 있습니다.

### PFADD / PFCOUNT

```java
// PFADD - 원소 추가, 반환값은 카디널리티 변화 여부 (변화 있으면 1, 없으면 0)
Long changed = opsForHyperLogLog().add("members", "123");  // → 1
opsForHyperLogLog().add("members", "500");
opsForHyperLogLog().add("members", "12");
opsForHyperLogLog().add("members", "12");  // 중복 → 카디널리티 변화 없음

// PFCOUNT - 고유 원소 수 추정
Long count = opsForHyperLogLog().size("members");  // → 3

// 여러 키의 합산 카디널리티 조회
Long total = opsForHyperLogLog().size("hll:1", "hll:2");
```

> **Redis CLI 참고**
> ```bash
> > PFADD members 123 → 1
> > PFADD members 500 → 1
> > PFADD members 12  → 1
> > PFCOUNT members   → 3
> ```

## 8. Geospatial

**경도/위도 데이터 쌍의 집합**입니다. 내부적으로 Sorted Set 기반으로 저장(geohash 인코딩)됩니다. `opsForGeo()`를 사용합니다.

### GEOADD

```java
// GEOADD travel longitude latitude member
opsForGeo().add("travel", new Point(14.39970, 50.09924), "prague");

// 여러 위치 한 번에 추가
opsForGeo().add("travel", new Point(127.0016985, 37.56421235), "seoul");
opsForGeo().add("travel", new Point(-122.434538, 37.785304), "SanFrancisco");
```

> `Point(x, y)` = `Point(경도, 위도)` 순서에 주의합니다.

### GEOPOS

저장된 경도/위도를 조회합니다. Redis는 geohash 인코딩으로 저장하므로 **미세한 부동소수점 오차**가 발생합니다.

```java
// GEOPOS travel prague
List<Point> positions = opsForGeo().position("travel", "prague", "seoul");
// positions.get(0) → Point(14.39969..., 50.09923...)  ← 미세한 오차 발생
```

### GEODIST

```java
// GEODIST travel seoul prague KM
Distance distance = opsForGeo().distance("travel", "seoul", "prague", Metrics.KILOMETERS);
System.out.println(distance.getValue()); // → 8252.9961

// 단위 변경: Metrics.KILOMETERS, Metrics.MILES, Metrics.NEUTRAL(m)
```

> **Redis CLI 참고**
> ```bash
> > GEOPOS travel prague
> 1) "14.39969927072525"
> 2) "50.099238974551746"
>
> > GEODIST travel seoul prague KM → "8252.9961"
> ```

## 9. Stream

**Redis를 메시지 브로커로 사용**할 수 있게 하는 자료구조입니다. Kafka의 영향을 받아 만들어졌으며 Producer/Consumer 개념이 존재합니다.

- 각 메시지는 자동 생성 ID(`타임스탬프-시퀀스`, 예: `1638125133152-0`)로 관리됩니다.
- 메시지를 소비해도 삭제되지 않습니다. (Kafka의 로그와 유사)

`opsForStream()`을 사용합니다.

### XADD — 메시지 추가 (Producer)

```java
// XADD mystream * name Alice action login
// * → ID 자동 생성
RecordId id = opsForStream().add(
    StreamRecords.newRecord()
        .in("mystream")
        .ofMap(Map.of("name", "Alice", "action", "login"))
);
System.out.println(id.getValue()); // → "1638125133152-0"
```

### XREAD — 메시지 읽기 (Consumer)

```java
// XREAD STREAMS mystream 0-0 → 처음부터 모든 메시지
List<MapRecord<String, Object, Object>> records =
    opsForStream().read(StreamOffset.create("mystream", ReadOffset.from("0-0")));

// 특정 ID 이후 메시지만
List<MapRecord<String, Object, Object>> after =
    opsForStream().read(StreamOffset.create("mystream", ReadOffset.from("1638125133152-0")));

records.forEach(record -> {
    System.out.println("ID: " + record.getId());
    System.out.println("Data: " + record.getValue());
});
```

| `ReadOffset` 값 | 의미 |
|---|---|
| `ReadOffset.from("0-0")` | 처음부터 모든 메시지 |
| `ReadOffset.from("특정ID")` | 해당 ID 이후 메시지 |
| `ReadOffset.latest()` (`$`) | 구독 시점 이후 새 메시지만 |

### XLEN

```java
// XLEN mystream - 스트림에 저장된 메시지 수
Long size = opsForStream().size("mystream");
```

## 10. 키 관리 커맨드

`RedisTemplate` 자체에서 키 수준의 관리 메서드를 제공합니다.

### 키 조회: KEYS vs SCAN

| | `keys()` | `scan()` |
|---|---|---|
| 내부 커맨드 | `KEYS` | `SCAN` |
| 방식 | 한 번에 전체 조회 | 커서 기반 분할 조회 |
| 블로킹 | O (싱글 스레드 블로킹) | X (Non-blocking) |
| 운영 환경 | **사용 지양** | 권장 |

```java
// KEYS - 운영 환경 사용 지양
Set<String> allKeys = redisTemplate.keys("*er*");

// SCAN - 커서 기반, try-with-resources 로 커서 반드시 닫기
ScanOptions options = ScanOptions.scanOptions().match("*er*").count(100).build();
try (Cursor<String> cursor = redisTemplate.scan(options)) {
    cursor.forEachRemaining(key -> System.out.println(key));
}

// TYPE 필터 - 특정 자료구조 타입의 키만 조회
ScanOptions typeFilter = ScanOptions.scanOptions().type("zset").build();
try (Cursor<String> cursor = redisTemplate.scan(typeFilter)) {
    cursor.forEachRemaining(key -> System.out.println(key));
}
```

각 자료구조 내부 순회도 동일한 방식으로 제공됩니다.

```java
opsForHash().scan(key, options);   // HSCAN
opsForSet().scan(key, options);    // SSCAN
opsForZSet().scan(key, options);   // ZSCAN
```

### 키 삭제: DEL vs UNLINK

| | `delete()` | `unlink()` |
|---|---|---|
| 내부 커맨드 | `DEL` | `UNLINK` |
| 방식 | 동기 삭제 | 비동기 삭제 |
| 메모리 해제 | 즉시 | 백그라운드 처리 |
| 대용량 키 | 블로킹 위험 | 권장 |

```java
// DEL - 동기 삭제, 반환값은 삭제된 키 수
Long deleted = redisTemplate.delete(List.of("key1", "key2"));

// UNLINK - 비동기 삭제, 대용량 키(큰 List, Hash 등) 삭제 시 권장
Long unlinked = redisTemplate.unlink(List.of("bigkey"));
```

### 만료 시간 설정

```java
// EXPIRE - Duration으로 만료 시간 설정
redisTemplate.expire("mykey", Duration.ofSeconds(100));

// EXPIREAT - 특정 시각에 만료
redisTemplate.expireAt("mykey", Instant.now().plusSeconds(300));

// TTL - 남은 초 반환 (-1: 미설정, -2: 키 없음)
Long ttl = redisTemplate.getExpire("mykey", TimeUnit.SECONDS);

// EXPIRETIME - 만료 유닉스 타임스탬프 반환 (Redis 7.0+)
// EXPIRE NX / XX / GT / LT 조건 옵션 (Redis 7.0+)
// → Spring Data Redis에서 직접 지원하지 않으므로 RedisCallback 사용
redisTemplate.execute((RedisCallback<Boolean>) conn -> {
    Object result = conn.execute("EXPIRE",
        "mykey".getBytes(), "100".getBytes(), "NX".getBytes());
    return Long.valueOf(1L).equals(result);
});
```

### 기타 커맨드

```java
// EXISTS
Boolean exists = redisTemplate.hasKey("mykey");

// TYPE - "string", "list", "hash", "set", "zset", "stream", "none"
String type = redisTemplate.type("mykey").code();

// RENAME / RENAMENX
redisTemplate.rename("oldkey", "newkey");
redisTemplate.renameIfAbsent("oldkey", "newkey"); // RENAMENX

// COPY
redisTemplate.copy("source", "dest", false);  // REPLACE false → dest 존재 시 실패
redisTemplate.copy("source", "dest", true);   // REPLACE true  → 덮어씀

// OBJECT ENCODING / IDLETIME - RedisCallback으로 처리
String encoding = redisTemplate.execute((RedisCallback<String>) conn -> {
    byte[] result = (byte[]) conn.execute("OBJECT", "ENCODING".getBytes(), "mykey".getBytes());
    return result != null ? new String(result) : null;
});

// SORT
List<String> sorted = redisTemplate.sort(SortQueryBuilder.sort("mylist").build());
List<String> alpha  = redisTemplate.sort(SortQueryBuilder.sort("mylist").alphabetical(true).build());
```

### FLUSHALL

```java
// 전체 삭제 (주의: 복구 불가)
redisTemplate.execute((RedisCallback<Void>) conn -> {
    conn.serverCommands().flushAll();
    return null;
});
```

## 정리

| 자료구조 | Operations | 특징 | 주요 활용 |
|---------|-----------|------|----------|
| String | `opsForValue()` | 단순 값, 원자적 숫자 조작 | 캐시, 카운터, 세션 |
| List | `opsForList()` | 순서 있는 목록 | 큐, 스택, 타임라인 |
| Hash | `opsForHash()` | 필드-값 객체 구조 | 사용자 정보, 상품 데이터 |
| Set | `opsForSet()` | 중복 없는 집합 | 태그, 팔로워, 집합 연산 |
| Sorted Set | `opsForZSet()` | 스코어 정렬 집합 | 랭킹, 우선순위 큐 |
| Bitmap | `opsForValue()` | 비트 단위 플래그 | 출석 체크, 기능 플래그 |
| HyperLogLog | `opsForHyperLogLog()` | 카디널리티 추정 | UV 카운팅 |
| Geospatial | `opsForGeo()` | 위치 정보 | 주변 장소 검색 |
| Stream | `opsForStream()` | 메시지 브로커 | 이벤트 로그, 실시간 처리 |
