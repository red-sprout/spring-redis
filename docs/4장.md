# Redis + MySQL 실전 활용 — 자료구조별 도메인 설계

> **시리즈**: Redis 학습 정리  
> **환경**: Spring Boot 4.0.5 / Spring Data Redis 4.0.4 / Java 21 / MySQL 8.x

---

## 들어가며

3장에서 Redis 자료구조의 기본 커맨드를 익혔다면, 4장은 **실제 애플리케이션에서 어떻게 쓰이는지**를 다룹니다.

핵심 원칙은 하나입니다.

> **MySQL은 영속(durability), Redis는 속도(performance)**

MySQL은 트랜잭션 보장이 필요한 데이터(회원, 게시글, 댓글 등)를 담당하고, Redis는 실시간 집계·조회가 필요한 데이터(리더보드, 좋아요 수, 읽지 않은 메시지 수 등)를 담당합니다. 두 저장소는 **서로 다른 트랜잭션 경계**를 가지므로 초기화 순서와 오류 처리에 주의해야 합니다.

이 장에서 다루는 패턴은 다음과 같습니다.

| 기능 | Redis 자료구조 | MySQL 역할 |
|------|--------------|-----------|
| 실시간 리더보드 | Sorted Set | — |
| 최근 검색 기록 | Sorted Set | — |
| 태그 기반 게시글 교집합 | Set | 태그·게시글 영속 |
| 댓글 좋아요 | Set | 댓글 영속 |
| 읽지 않은 메시지 수 | Hash | 채널 영속 |
| DAU (일별 접속자 수) | Bitmap | — |
| API 유니크 호출 미터링 | HyperLogLog | — |
| 위치 기반 검색 | Geo Set | 장소 영속 |

## 1. 프로젝트 구조와 초기화 전략

### 패키지 구조

```
domain/
└── ch4/
    ├── user/          # 회원 (MySQL)
    ├── leaderboard/   # 리더보드 (Redis Sorted Set)
    ├── search/        # 최근 검색 기록 (Redis Sorted Set)
    ├── post/          # 게시글 + 태그 (MySQL + Redis Set)
    ├── comment/       # 댓글 + 좋아요 (MySQL + Redis Set)
    ├── chat/          # 읽지 않은 메시지 (MySQL + Redis Hash)
    ├── analytics/     # DAU + 미터링 (Redis Bitmap + HyperLogLog)
    ├── location/      # 위치 검색 (MySQL + Redis Geo)
    └── seed/          # 초기 데이터 조회 (Swagger 테스트용)
global/
└── init/
    ├── DataInitService.java    # MySQL 초기 데이터 (단일 @Transactional)
    └── DataInitializer.java    # ApplicationRunner — MySQL 커밋 후 Redis 초기화
```

### MySQL vs Redis 트랜잭션 분리

MySQL의 `@Transactional`은 Redis에 적용되지 않습니다. 따라서 초기화 코드를 아무 생각 없이 작성하면 MySQL이 실패했을 때 일부 데이터만 저장된 상태가 됩니다.

**잘못된 패턴 — 각 서비스 호출이 별개 트랜잭션**

```java
// ❌ MySQL 삽입이 중간에 실패해도 앞서 커밋된 데이터는 롤백되지 않음
userService.save("player101");   // 커밋
userService.save("player143");   // 커밋
postService.createPost(...);     // 여기서 실패해도 앞의 user는 이미 저장됨
```

**올바른 패턴 — MySQL 전체를 단일 트랜잭션으로 묶기**

```java
// DataInitService.java
@Transactional
public InitResult initMysql() {
    User u1 = userRepository.save(...);
    User u2 = userRepository.save(...);
    Post p1 = postRepository.save(...);
    // ... 전부 한 트랜잭션 — 실패 시 전체 롤백
    return new InitResult(users, tags, posts, comments, channels, places);
}
```

```java
// DataInitializer.java (ApplicationRunner)
public void run(ApplicationArguments args) {
    // 1. MySQL 전체를 원자적으로 커밋
    InitResult result = dataInitService.initMysql();

    // 2. MySQL 커밋 이후 Redis 작업 (별도 처리)
    leaderboardService.addScore(...);
    analyticsService.recordVisit(...);
    locationService.syncAllToRedis();   // MySQL → Redis 동기화
    // ...
}
```

> **핵심**: MySQL과 Redis는 같은 트랜잭션에 묶일 수 없습니다. MySQL 커밋이 완전히 끝난 뒤 Redis 작업을 수행하세요.

## 3. 실시간 리더보드 — Sorted Set

게임·커머스의 순위표는 **ZADD + ZREVRANGE** 조합으로 구현합니다. Redis Sorted Set은 score 기반으로 항상 정렬된 상태를 유지하므로, 별도의 정렬 연산 없이 O(log N) 삽입과 O(log N + M) 범위 조회가 가능합니다.

### 키 설계

```
daily-score:{yyyyMMdd}   # 일별 점수
weekly-score:{weekKey}   # 주간 합산 점수 (ZUNIONSTORE 결과)
```

### 점수 등록 / 갱신

```java
// ZADD daily-score:260413 28 player:5
redisTemplate.opsForZSet().add(
    "daily-score:" + date,
    "player:" + userId,
    score
);

// ZINCRBY daily-score:260413 10 player:5  → 기존 score에 누적 가산
Double newScore = redisTemplate.opsForZSet().incrementScore(
    "daily-score:" + date,
    "player:" + userId,
    delta
);
```

> **Redis CLI 참고**
> ```bash
> > ZADD daily-score:260413 28 player:286
> > ZADD daily-score:260413 400 player:234
> > ZINCRBY daily-score:260413 50 player:286   # → 78
> ```

### 상위 N명 조회

```java
// ZREVRANGE daily-score:260413 0 4 WITHSCORES → 상위 5명 (내림차순)
Set<ZSetOperations.TypedTuple<Object>> result =
    redisTemplate.opsForZSet()
        .reverseRangeWithScores("daily-score:" + date, 0, n - 1);

List<PlayerScore> top = result.stream()
    .map(t -> new PlayerScore((String) t.getValue(), t.getScore()))
    .toList();
```

### 내 순위 / 점수 조회

```java
// ZREVRANK → 0-based 역순 순위 (1위 = 0)
Long rank = redisTemplate.opsForZSet()
    .reverseRank("daily-score:" + date, "player:" + userId);

// ZSCORE
Double score = redisTemplate.opsForZSet()
    .score("daily-score:" + date, "player:" + userId);
```

### 주간 점수 합산 (ZUNIONSTORE)

여러 날짜의 일별 Sorted Set을 합산해 주간 순위를 만듭니다. 동일 멤버의 score는 **합산**됩니다.

```java
// ZUNIONSTORE weekly-score:week1 3 daily-score:260413 daily-score:260414 daily-score:260415
List<String> dailyKeys = dates.stream()
    .map(d -> "daily-score:" + d)
    .toList();

redisTemplate.opsForZSet().unionAndStore(
    dailyKeys.get(0),
    dailyKeys.subList(1, dailyKeys.size()),
    "weekly-score:" + weekKey
);
```

> **Redis CLI 참고**
> ```bash
> > ZUNIONSTORE weekly-score:w1 3 daily-score:260406 daily-score:260407 daily-score:260408
> > ZREVRANGE weekly-score:w1 0 -1 WITHSCORES
> ```

## 4. 최근 검색 기록 — Sorted Set + 고정 크기

검색 기록은 **시간(Unix ms)을 score**로 저장해 최신순 정렬을 구현합니다. `ZADD` + `ZREMRANGEBYRANK` 조합으로 최대 N개를 유지합니다.

### 키 설계

```
search-keyword:{userId}
```

### 키워드 추가 (최대 5개 유지)

```java
// ZADD search-keyword:{userId} {currentTimeMs} {keyword}
redisTemplate.opsForZSet().add(key, keyword, System.currentTimeMillis());

// ZREMRANGEBYRANK 0 -6  → score 오름차순(오래된 것)으로 6번째 이상 전부 제거
// 즉, 최신 5개만 남김
redisTemplate.opsForZSet().removeRange(key, 0, -(MAX_KEYWORDS + 1));
```

> **인덱스 계산 이해하기**  
> `removeRange(key, 0, -6)` 에서 `-6`은 뒤에서 6번째 (= 총 개수 - 5번째)입니다.  
> 5개가 이미 있을 때 하나를 추가하면 6개가 되고, 이 중 가장 오래된 1개(인덱스 0)가 제거됩니다.

### 최근 검색어 조회

```java
// ZREVRANGE search-keyword:{userId} 0 -1 WITHSCORES → 최신순
Set<ZSetOperations.TypedTuple<Object>> result =
    redisTemplate.opsForZSet()
        .reverseRangeWithScores("search-keyword:" + userId, 0, -1);
```

### 검색어 삭제 / 전체 삭제

```java
// ZREM — 특정 키워드만 삭제
redisTemplate.opsForZSet().remove("search-keyword:" + userId, keyword);

// DEL — 검색 기록 전체 삭제
redisTemplate.delete("search-keyword:" + userId);
```

> **Redis CLI 참고**
> ```bash
> > ZADD search-keyword:1 1744500001000 실버
> > ZADD search-keyword:1 1744500002000 기모후드
> > ZREVRANGE search-keyword:1 0 -1 WITHSCORES
> > ZREMRANGEBYRANK search-keyword:1 0 -6
> ```

## 5. 태그 교집합 게시글 조회 — Set

"의류 AND 아우터 태그가 붙은 게시글"처럼 **여러 조건의 교집합**이 필요한 조회는 Redis Set의 `SINTER`로 처리합니다. MySQL JOIN 없이 O(N × M) 수준의 빠른 교집합 연산이 가능합니다.

### 키 설계

```
tag:{tagId}   → 해당 태그가 붙은 postId 목록 (Set)
```

MySQL에 PostTag를 저장할 때 Redis Set에도 함께 동기화합니다.

```java
// MySQL 저장
postTagRepository.save(PostTag.builder().post(post).tag(tag).build());

// Redis Set 동기화: SADD tag:{tagId} {postId}
redisTemplate.opsForSet().add("tag:" + tagId, postId.toString());
```

### 교집합 조회

```java
// SINTER tag:{id1} tag:{id2} ... → 두 태그 모두 붙은 postId 집합
String[] keys = tagIds.stream()
    .map(id -> "tag:" + id)
    .toArray(String[]::new);

Set<Object> postIdSet = redisTemplate.opsForSet()
    .intersect(keys[0], List.of(keys).subList(1, keys.length));

// postId로 MySQL 조회
List<Long> postIds = postIdSet.stream()
    .map(o -> Long.parseLong((String) o))
    .toList();
List<Post> posts = postRepository.findAllById(postIds);
```

> **Redis CLI 참고**
> ```bash
> > SADD tag:1 101 103    # 의류 태그 → 게시글 101, 103
> > SADD tag:3 103        # 아우터 태그 → 게시글 103
> > SINTER tag:1 tag:3    # → {103}
> ```

> **왜 Redis Set?**  
> MySQL에서 `SELECT ... WHERE tag_id IN (1,3) GROUP BY post_id HAVING COUNT(*) = 2` 같은 쿼리도 가능하지만, 태그 수·게시글 수가 많아질수록 인덱스 스캔 비용이 커집니다. Redis Set 교집합은 인메모리 비트 연산으로 수십만 건도 밀리초 내에 처리합니다.

## 6. 댓글 좋아요 — Set

좋아요는 두 가지 요구사항이 동시에 있습니다.
1. **중복 방지** — 같은 유저가 두 번 좋아요를 누를 수 없음
2. **빠른 카운팅** — 좋아요 수를 즉시 반환

Redis Set은 원소의 유일성을 보장하므로 두 조건을 자연스럽게 만족합니다.

### 키 설계

```
comment-like:{commentId}   → 좋아요한 userId 목록 (Set)
```

### 좋아요 / 취소

```java
// SADD comment-like:{commentId} {userId}  → 이미 존재하면 무시
redisTemplate.opsForSet().add("comment-like:" + commentId, userId.toString());

// SREM comment-like:{commentId} {userId}
redisTemplate.opsForSet().remove("comment-like:" + commentId, userId.toString());
```

### 좋아요 수 / 확인 / 목록

```java
// SCARD — 좋아요 총 개수
Long count = redisTemplate.opsForSet().size("comment-like:" + commentId);

// SISMEMBER — 특정 유저가 좋아요했는지 O(1) 확인
Boolean liked = redisTemplate.opsForSet()
    .isMember("comment-like:" + commentId, userId.toString());

// SMEMBERS — 좋아요한 유저 목록 전체
Set<Object> users = redisTemplate.opsForSet()
    .members("comment-like:" + commentId);
```

> **Redis CLI 참고**
> ```bash
> > SADD comment-like:12554 345 25 967
> > SADD comment-like:12554 345    # 중복 → 무시 (SCARD 변화 없음)
> > SCARD comment-like:12554       # → 3
> > SISMEMBER comment-like:12554 25  # → 1 (좋아요 함)
> > SISMEMBER comment-like:12554 999 # → 0 (좋아요 안 함)
> ```

## 7. 읽지 않은 메시지 수 — Hash

채팅에서 유저별·채널별 읽지 않은 메시지 수를 관리합니다. `HINCRBY`를 사용해 메시지가 올 때마다 카운터를 원자적으로 증가시킵니다.

### 키 설계

```
user:{userId}
  field: channel:{channelId}  → 읽지 않은 메시지 수 (정수 문자열)
```

### 중요: StringRedisTemplate 사용

`HINCRBY`는 필드 값이 **정수 문자열**(`"0"`, `"27"`)이어야 동작합니다.  
`RedisTemplate<String, Object>`에 JSON 직렬화를 설정하면 `"0"`이 `"\"0\""`으로 저장되어 `HINCRBY`가 `ERR hash value is not an integer` 오류를 냅니다.

정수 카운터 조작에는 반드시 `StringRedisTemplate`을 사용합니다.

```java
// ❌ JSON 직렬화 → HINCRBY 오류
redisTemplate.opsForHash().put(key, field, "0");  // 저장: "\"0\""

// ✅ StringRedisTemplate → HINCRBY 정상 동작
stringRedisTemplate.opsForHash().put(key, field, "0");  // 저장: "0"
```

### 채널 참여 (HSETNX)

이미 참여한 채널이면 카운터를 초기화하지 않도록 `putIfAbsent`를 사용합니다.

```java
// HSETNX user:{userId} channel:{channelId} 0
// 필드가 없을 때만 "0" 으로 초기화 → 이미 있으면 무시
stringRedisTemplate.opsForHash().putIfAbsent(
    "user:" + userId,
    "channel:" + channelId,
    "0"
);
```

### 메시지 수신 (HINCRBY)

```java
// HINCRBY user:{userId} channel:{channelId} 1
Long unread = stringRedisTemplate.opsForHash().increment(
    "user:" + userId,
    "channel:" + channelId,
    1
);
```

### 메시지 읽음 처리

```java
// HSET user:{userId} channel:{channelId} 0  → 카운터 초기화
stringRedisTemplate.opsForHash().put(
    "user:" + userId,
    "channel:" + channelId,
    "0"
);
```

### 전체 / 특정 채널 조회

```java
// HGETALL user:{userId} → 모든 채널의 읽지 않은 수
Map<Object, Object> all = stringRedisTemplate.opsForHash()
    .entries("user:" + userId);

// HGET user:{userId} channel:{channelId}
Object count = stringRedisTemplate.opsForHash()
    .get("user:" + userId, "channel:" + channelId);
```

> **Redis CLI 참고**
> ```bash
> > HSETNX user:123 channel:4234 0    # 참여
> > HINCRBY user:123 channel:4234 1   # 메시지 수신 → 1
> > HINCRBY user:123 channel:4234 1   # → 2
> > HSET user:123 channel:4234 0      # 읽음 처리 → 0
> > HGETALL user:123
> # channel:4234 = 0
> # channel:3135 = 27
> # channel:1239 = 128
> ```

## 8. DAU 집계 — Bitmap

Bitmap은 userId를 **비트 오프셋**으로 사용해 방문 여부를 1비트로 기록합니다. 1억 유저도 약 12MB로 표현할 수 있어 공간 효율이 매우 높습니다.

### 키 설계

```
uv:{yyyyMMdd}   → 날짜별 방문자 비트맵
```

### 방문 기록

```java
// SETBIT uv:260413 {userId} 1
stringRedisTemplate.opsForValue().setBit("uv:" + date, userId, true);
```

### DAU 조회 (BITCOUNT)

`BITCOUNT`는 `opsForValue()`에서 직접 지원하지 않으므로 `RedisCallback`으로 로우레벨 커맨드를 실행합니다.

```java
// BITCOUNT uv:260413
Long dau = stringRedisTemplate.execute((RedisCallback<Long>) connection ->
    connection.stringCommands().bitCount(
        ("uv:" + date).getBytes(StandardCharsets.UTF_8)
    )
);
```

### 방문 여부 확인

```java
// GETBIT uv:260413 {userId}
Boolean visited = stringRedisTemplate.opsForValue()
    .getBit("uv:" + date, userId);
```

### 연속 접속자 수 (BITOP AND)

여러 날짜 비트맵의 AND 연산으로 **모든 날짜에 접속한 유저 수**를 구합니다.

```java
// BITOP AND retained:260413-260414-260415 uv:260413 uv:260414 uv:260415
byte[][] sourceKeys = dates.stream()
    .map(d -> ("uv:" + d).getBytes(StandardCharsets.UTF_8))
    .toArray(byte[][]::new);

stringRedisTemplate.execute((RedisCallback<Void>) connection -> {
    connection.stringCommands().bitOp(
        RedisStringCommands.BitOperation.AND,
        "retained:...".getBytes(StandardCharsets.UTF_8),
        sourceKeys
    );
    return null;
});

// 결과 키에 BITCOUNT → 연속 접속 유저 수
Long retained = stringRedisTemplate.execute((RedisCallback<Long>) connection ->
    connection.stringCommands().bitCount(destKey.getBytes())
);
```

> **Redis CLI 참고**
> ```bash
> > SETBIT uv:260413 14 1
> > SETBIT uv:260413 3  1
> > SETBIT uv:260413 7  1
> > BITCOUNT uv:260413         # → 3 (DAU = 3)
> > BITOP AND retained uv:260413 uv:260414
> > BITCOUNT retained          # 양일 모두 접속한 유저 수
> ```

> **왜 StringRedisTemplate?**  
> `SETBIT`·`BITCOUNT`·`BITOP`은 문자열 명령 계열입니다. `RedisTemplate<String, Object>`에 JSON 직렬화를 쓰면 비트맵 키 내부 데이터를 JSON 래퍼가 손상시킵니다. 비트맵·카운터는 `StringRedisTemplate`을 전용으로 사용하세요.

## 9. API 유니크 미터링 — HyperLogLog

HyperLogLog는 수백만 건 데이터도 **약 12KB 메모리**로 유니크 카운팅을 수행합니다. 오차율은 약 0.81%로, 정확한 숫자보다 **대략적인 규모**가 필요한 미터링·통계에 적합합니다.

### 키 설계

```
api-meter:{userId}   → 유저별 유니크 API 호출 로그 ID 집합
```

### 호출 기록

```java
// PFADD api-meter:{userId} {logId}
redisTemplate.opsForHyperLogLog().add("api-meter:" + userId, logId);
```

### 유니크 호출 수 조회

```java
// PFCOUNT api-meter:{userId}
Long count = redisTemplate.opsForHyperLogLog().size("api-meter:" + userId);
```

> **Redis CLI 참고**
> ```bash
> > PFADD api-meter:1 log-1 log-2 log-3
> > PFADD api-meter:1 log-1          # 중복 → 카운트 변화 없음
> > PFCOUNT api-meter:1              # → 3
> ```

> **HyperLogLog vs Set 비교**
>
> | | Set | HyperLogLog |
> |---|---|---|
> | 정확도 | 100% | ~99.2% |
> | 메모리 | O(N) | 고정 12KB |
> | 원소 조회 | SMEMBERS 가능 | 불가 |
> | 적합한 용도 | 소규모 정확한 집계 | 대규모 근사 집계 |

## 10. 위치 기반 검색 — Geo Set

Redis Geo Set은 위경도 좌표를 내부적으로 **geohash** 인코딩으로 Sorted Set에 저장합니다. `GEOSEARCH` 커맨드로 특정 좌표 반경 내 장소를 거리순으로 조회할 수 있습니다.

### 키 설계

```
places   → 장소 이름을 멤버로, geohash를 score로 저장
```

MySQL에는 장소 영속 데이터를, Redis에는 좌표 인덱스를 분리 저장합니다.

### 장소 등록

```java
// GEOADD places {longitude} {latitude} {name}
redisTemplate.opsForGeo().add("places",
    new Point(longitude, latitude),
    name
);
```

MySQL에 이미 저장된 장소를 Redis로 일괄 동기화:

```java
// MySQL → Redis 전체 동기화
placeRepository.findAll().forEach(p ->
    redisTemplate.opsForGeo().add("places",
        new Point(p.getLongitude(), p.getLatitude()),
        p.getName()
    )
);
```

### 좌표로 반경 검색 (GEOSEARCH FROMLONLAT)

Spring Data Redis 4.x에서는 `GeoReference.fromCoordinate()`를 사용합니다. (3.x의 `fromCoordinates()`에서 이름이 변경됨)

```java
// GEOSEARCH places FROMLONLAT 127.0276 37.4979 BYRADIUS 5 KM ASC COUNT 3
GeoResults<RedisGeoCommands.GeoLocation<Object>> results =
    redisTemplate.opsForGeo().search(
        "places",
        GeoReference.fromCoordinate(new Point(longitude, latitude)),
        new Distance(radiusKm, Metrics.KILOMETERS),
        RedisGeoCommands.GeoSearchCommandArgs.newGeoSearchArgs()
            .sortAscending()
            .limit(count)
    );
```

### 장소명으로 반경 검색 (GEOSEARCH FROMMEMBER)

```java
// GEOSEARCH places FROMMEMBER 강남역 BYRADIUS 5 KM ASC COUNT 5
GeoResults<RedisGeoCommands.GeoLocation<Object>> results =
    redisTemplate.opsForGeo().search(
        "places",
        GeoReference.fromMember(placeName),
        new Distance(radiusKm, Metrics.KILOMETERS),
        RedisGeoCommands.GeoSearchCommandArgs.newGeoSearchArgs()
            .sortAscending()
            .limit(count)
    );
```

### 두 장소 간 거리

```java
// GEODIST places 서울역 강남역 KM
Distance dist = redisTemplate.opsForGeo()
    .distance("places", "서울역", "강남역", Metrics.KILOMETERS);
```

### 좌표 조회

```java
// GEOPOS places 홍대입구역
List<Point> positions = redisTemplate.opsForGeo()
    .position("places", "홍대입구역");
```

> **Redis CLI 참고**
> ```bash
> > GEOADD places 127.0276 37.4979 강남역
> > GEOADD places 126.9707 37.5546 서울역
> > GEODIST places 서울역 강남역 KM          # → 약 7.18
> > GEOSEARCH places FROMLONLAT 127.0276 37.4979 BYRADIUS 5 KM ASC COUNT 3
> ```

> **Spring Data Redis 4.x API 변경 사항**  
>
> | 변경 전 (3.x) | 변경 후 (4.x) |
> |---|---|
> | `GeoReference.fromCoordinates()` | `GeoReference.fromCoordinate()` |
> | `GeoSearchCommandArgs.asc()` | `.sortAscending()` |
> | `GeoSearchCommandArgs.count()` | `.limit()` |

## 정리

| 도메인 | 자료구조 | 핵심 커맨드 | Redis 선택 이유 |
|--------|---------|------------|---------------|
| 리더보드 | Sorted Set | ZADD, ZREVRANGE, ZINCRBY, ZUNIONSTORE | score 기반 자동 정렬, O(log N) 삽입 |
| 검색 기록 | Sorted Set | ZADD, ZREVRANGE, ZREMRANGEBYRANK | 시간을 score로 → 최신순, 최대 N개 자동 관리 |
| 태그 교집합 | Set | SADD, SINTER | 인메모리 집합 연산, 중복 자동 제거 |
| 좋아요 | Set | SADD, SREM, SCARD, SISMEMBER | 중복 방지 + O(1) 카운팅 |
| 읽지 않은 메시지 | Hash | HSETNX, HINCRBY, HSET, HGETALL | 유저 단위 채널별 카운터 묶음 |
| DAU | Bitmap | SETBIT, BITCOUNT, BITOP | 1비트/유저, 1억 유저 = 12MB |
| API 미터링 | HyperLogLog | PFADD, PFCOUNT | 대규모 유니크 카운팅, 고정 12KB |
| 위치 검색 | Geo Set | GEOADD, GEOSEARCH, GEODIST | geohash 인덱싱, 반경 쿼리 내장 |
