# Redis 파이프라이닝 — 대용량 키 분석으로 보는 성능 차이

> **시리즈**: Redis 학습 정리  
> **환경**: Spring Boot 4.0.5 / Spring Data Redis 4.0.4 / Java 21 / Docker Redis (localhost:6379)

---

## 들어가며

Redis는 싱글 스레드로 명령을 처리하지만, 클라이언트와 서버 사이의 **네트워크 왕복(Round Trip)**이 병목이 되는 경우가 많습니다. 키 하나당 `TYPE`, `MEMORY USAGE`를 각각 조회하면, 키 수 × 2번의 왕복이 발생합니다. 10만 개 키라면 **20만 번**입니다.

**파이프라이닝(Pipelining)** 은 여러 명령을 한 번에 묶어 전송하고 결과를 일괄로 수신하는 방식입니다. 왕복 횟수를 `배치 수 × 2`로 줄여, 대용량 환경에서 극적인 속도 차이를 만들어냅니다.

이 장에서는 **10만 개 키의 타입과 메모리 사용량을 집계**하는 두 가지 방식을 비교합니다.

```
[For 루프]  각 키마다 TYPE → MEMORY USAGE 순으로 개별 요청   →  31.5초
[파이프라인] 배치(1000개) 단위로 TYPE × 1000 + MEMORY USAGE × 1000 일괄 전송  →  2.1초
```

## 1. 파이프라인 개념

### 일반 요청 vs 파이프라인

```
[일반 요청]
Client ──TYPE key1──▶ Redis
Client ◀──"string"── Redis
Client ──MEMORY USAGE key1──▶ Redis
Client ◀──56── Redis
Client ──TYPE key2──▶ Redis
...
왕복: 키 수 × 명령 수

[파이프라인]
Client ──TYPE key1──▶
        ──TYPE key2──▶
        ──TYPE key3──▶  Redis (일괄 처리)
        ──MEMORY USAGE key1──▶
        ──MEMORY USAGE key2──▶
        ──MEMORY USAGE key3──▶
Client ◀── 결과 6개 일괄 수신 ──
왕복: 배치 수 × 명령 수
```

### 파이프라인의 특성

| 구분 | 내용 |
|---|---|
| 원자성 | **없음** — 명령 사이에 다른 클라이언트 명령이 끼어들 수 있음 |
| 순서 보장 | **있음** — 결과 순서는 전송 순서와 동일 |
| 에러 처리 | 한 명령이 실패해도 나머지는 계속 실행됨 |
| 적합한 상황 | 서로 독립적인 명령을 대량으로 보낼 때 |

> 원자성이 필요하다면 파이프라인 대신 **트랜잭션(`MULTI`/`EXEC`)** 또는 **Lua 스크립트**를 사용해야 합니다.

### Spring Data Redis에서 파이프라인 사용하기

`executePipelined(RedisCallback)` 안에서 명령을 쌓으면, 콜백이 종료될 때 한 번에 전송됩니다. 콜백 내부의 각 명령은 **항상 `null`을 반환**하며, 실제 결과는 `executePipelined()`의 반환값인 `List<Object>`에 순서대로 담깁니다.

```java
List<Object> results = stringRedisTemplate.executePipelined((RedisCallback<Object>) conn -> {
    conn.stringCommands().set("name".getBytes(), "Redi".getBytes()); // null 반환
    conn.opsForValue().increment("counter");                         // null 반환
    conn.stringCommands().get("name".getBytes());                    // null 반환
    return null; // 콜백 자체도 반드시 null 반환
});

// results = [true, 1L, "Redi"]  ← 명령 순서대로
```

## 4. 코드 설명

### 4.1 테스트 데이터 생성

데이터 삽입 자체도 파이프라인을 사용해 빠르게 처리합니다. 1,000개 단위 배치로 String 10만 개를 삽입합니다.

```java
// TestDataService.java
private void insertStrings() {
    for (int i = 0; i < STRING_COUNT; i += BATCH_SIZE) {
        final int from = i;
        final int to   = Math.min(i + BATCH_SIZE, STRING_COUNT);
        stringRedisTemplate.executePipelined((RedisCallback<Object>) conn -> {
            for (int j = from; j < to; j++) {
                conn.stringCommands().set(
                    ("ch12:str:" + j).getBytes(UTF_8),
                    ("value-" + j).getBytes(UTF_8)
                );
            }
            return null;
        });
    }
}
```

### 4.2 For 루프 방식

키마다 `TYPE`, `MEMORY USAGE`를 **각각 개별 호출**합니다. 직관적이지만 네트워크 왕복이 `키 수 × 2`번 발생합니다.

```java
// ForLoopAnalysisService.java
public AnalysisResult analyze() {
    List<String> allKeys = scanAllKeys();   // SCAN으로 전체 키 수집

    for (String key : allKeys) {
        // 왕복 1: TYPE key
        DataType type = stringRedisTemplate.type(key);

        // 왕복 2: MEMORY USAGE key (Lua 스크립트로 호출 — 이유는 트러블슈팅 참고)
        Object mem = stringRedisTemplate.execute((RedisCallback<Object>) conn ->
            conn.scriptingCommands().eval(MEMORY_USAGE_SCRIPT, ReturnType.INTEGER, 1,
                key.getBytes(UTF_8)));
        ...
    }
}
```

**SCAN으로 전체 키 조회**

`KEYS *`는 프로덕션에서 절대 사용하면 안 됩니다 (싱글 스레드 블로킹). `SCAN`은 커서 기반으로 조금씩 읽어 서버 부하를 분산합니다.

```java
// RedisTemplate.scan()은 Cursor<String>을 반환 — byte[] 변환 불필요
private List<String> scanAllKeys() {
    List<String> keys = new ArrayList<>();
    try (Cursor<String> cursor = stringRedisTemplate.scan(
            ScanOptions.scanOptions().count(1000).build())) {
        cursor.forEachRemaining(keys::add);
    }
    return keys;
}
```

### 4.3 파이프라인 방식

배치(1,000개) 단위로 `TYPE × N + MEMORY USAGE × N`을 **단일 파이프라인**으로 묶어 전송합니다.

```java
// PipelineAnalysisService.java
for (int i = 0; i < allKeys.size(); i += BATCH_SIZE) {
    List<String> batch = allKeys.subList(i, Math.min(i + BATCH_SIZE, allKeys.size()));
    int n = batch.size();

    List<Object> results = stringRedisTemplate.executePipelined((RedisCallback<Object>) conn -> {
        // 앞쪽 n개: TYPE 명령
        for (String key : batch) {
            conn.keyCommands().type(key.getBytes(UTF_8));
        }
        // 뒤쪽 n개: MEMORY USAGE 명령
        for (String key : batch) {
            conn.scriptingCommands().eval(MEMORY_USAGE_SCRIPT, ReturnType.INTEGER, 1,
                key.getBytes(UTF_8));
        }
        return null;
    });

    // results[0..n-1]   → DataType  (TYPE 결과)
    // results[n..2n-1]  → Long      (MEMORY USAGE 결과)
    for (int j = 0; j < n; j++) {
        DataType type = (DataType) results.get(j);
        Object   mem  = results.get(n + j);
        ...
    }
}
```

**왕복 횟수 비교**

| 방식 | 왕복 횟수 (키 101,126개 기준) |
|---|---|
| For 루프 | 101,126 × 2 = **202,252번** |
| 파이프라인 (배치 1,000) | ceil(101,126 / 1,000) × 2 = **204번** |

## 6. 트러블슈팅

### 이슈 1 — `memoryUsage()` 메서드가 존재하지 않음

**현상**: 컴파일 에러
```
error: cannot find symbol
    conn.keyCommands().memoryUsage(key.getBytes(...))
                       ^
```

**원인**: Spring Data Redis 2.x에 있던 `RedisKeyCommands.memoryUsage(byte[])` 메서드가 4.0.x에서 제거됨.

```bash
# jar에서 직접 확인
javap org.springframework.data.redis.connection.RedisKeyCommands
# → memoryUsage 메서드 없음
```

**해결**: `conn.execute("MEMORY", "USAGE", key)` 시도 → 다른 문제 발생 (이슈 2 참고). 최종적으로 `scriptingCommands().eval()` 사용.

---

### 이슈 2 — `ByteArrayOutput does not support set(long)`

**현상**: 런타임 에러
```
java.lang.UnsupportedOperationException:
    io.lettuce.core.output.ByteArrayOutput does not support set(long)
```

**원인**: `conn.execute(String command, byte[]... args)`는 Lettuce 내부에서 응답을 **`ByteArrayOutput`** 으로 수신합니다. 그런데 `MEMORY USAGE`는 바이트 배열이 아닌 **정수(Long)** 를 반환하므로, `set(long)` 호출 시 `UnsupportedOperationException`이 발생합니다.

```
Redis Response: (integer) 56
                    ↓
Lettuce가 ByteArrayOutput.set(long 56) 시도
                    ↓
UnsupportedOperationException ← byte[]에 long을 담을 수 없음
```

**해결**: `conn.scriptingCommands().eval()`에 `ReturnType.INTEGER`를 지정하면 Lettuce가 **`IntegerOutput(LongOutput)`** 을 사용해 정수 응답을 올바르게 처리합니다. 파이프라인 내부에서도 동일하게 동작합니다.

```java
private static final byte[] MEMORY_USAGE_SCRIPT =
    "return redis.call('MEMORY', 'USAGE', KEYS[1])".getBytes(UTF_8);

// ReturnType.INTEGER → Lettuce가 LongOutput 사용 → Long 정상 반환
conn.scriptingCommands().eval(MEMORY_USAGE_SCRIPT, ReturnType.INTEGER, 1, key.getBytes(UTF_8));
```

| 방법 | Lettuce Output | 결과 |
|---|---|---|
| `conn.execute("MEMORY", "USAGE", key)` | `ByteArrayOutput` | `UnsupportedOperationException` |
| `conn.scriptingCommands().eval(..., ReturnType.INTEGER, ...)` | `LongOutput` | `Long` 정상 반환 |

---

### 이슈 3 — `Cursor<byte[]>` vs `Cursor<String>` SCAN 처리

**현상**: 초기 구현에서 `RedisCallback` 안에 SCAN 로직을 직접 작성했으나 코드가 복잡함.

```java
// 복잡한 방법 (byte[] 수동 변환 + IOException 처리)
stringRedisTemplate.execute((RedisCallback<Void>) conn -> {
    try (Cursor<byte[]> cursor = conn.scan(opts)) {
        while (cursor.hasNext()) {
            keys.add(new String(cursor.next(), UTF_8)); // byte[] → String 변환
        }
    } catch (IOException e) {
        throw new RuntimeException("SCAN failed", e);
    }
    return null;
});
```

**해결**: `RedisTemplate.scan(ScanOptions)`은 `Cursor<K>`를 반환합니다. `StringRedisTemplate`에서는 `Cursor<String>`이므로 byte 변환과 예외 처리가 모두 불필요합니다.

```java
// 깔끔한 방법 — try-with-resources로 커서 자동 close
try (Cursor<String> cursor = stringRedisTemplate.scan(
        ScanOptions.scanOptions().count(1000).build())) {
    cursor.forEachRemaining(keys::add);
}
```
